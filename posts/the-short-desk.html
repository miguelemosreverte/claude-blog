The Short Desk

An AI trading agent that shorts Bitcoin on Deribit,
with backtesting rigor and an event-sourced audit trail

Act I

The Thesis

MicroStrategy buys Bitcoin. It has made this its corporate identity: accumulate BTC, hold indefinitely, repeat. The Anti-MicroStrategy does the opposite. It shorts Bitcoin on Deribit perpetual futures, guided by a bearish thesis and an AI agent that reads the market before every trade.

The agent is Claude. Not a fine-tuned model, not a statistical classifier, not a rule engine—a language model invoked via the Claude Code CLI. Every 15 minutes, the system collects market data from five sources: CoinGecko for spot prices, the Deribit API for futures and funding rates, FRED for macroeconomic indicators, a fear-and-greed index for sentiment, and computed technical indicators (RSI, MACD, Bollinger Bands). This data is assembled into a structured prompt. Claude reads it and returns a decision: SHORT, HOLD, CLOSE, REDUCE, or INCREASE_SHORT.

The trade is then executed on Deribit. The analysis, decision, and outcome are stored in SQLite. A WSJ-styled HTML report is generated and opens in the browser. Every cycle is documented. Every decision is traceable.

Act II

The Pipeline

The architecture is a loop, not a service. Every 15 minutes, a Python script wakes up, collects fresh data, calls Claude, executes the resulting trade, and sleeps. There is no web server, no message queue, no orchestration layer. The complexity lives in the data aggregation and in the backtesting framework.

The data sources were chosen for orthogonality. Spot price and funding rate measure what the market is doing. RSI, MACD, and Bollinger Bands measure what it has been doing. Fear-and-greed measures what the crowd feels. FRED data (federal funds rate, CPI, Treasury spreads) measures the macro environment the market exists in. Claude sees all four dimensions simultaneously.

Act III

The Backtest

Anyone can write a trading bot. The question is whether it works. The backtesting framework uses sliding-window cross-validation borrowed from machine learning—the same technique used to validate time-series models, applied to an AI trading agent. 

// Sliding-window cross-validation: 30 days of hourly candles

Fold 1: [===== 25d train =====][== 5d test ==]
Fold 2:   [===== 25d train =====][== 5d test ==]
Fold 3:     [===== 25d train =====][== 5d test ==]

Each fold: Claude sees 25 days of history,
makes decisions for the 5-day test window,
PnL is simulated against actual prices.
Alpha = agent return - buy-and-hold return.

The key metric is alpha—the return above or below what a simple buy-and-hold would have achieved. A positive alpha means the agent is adding value beyond the market's natural movement. The backtest measures this across multiple non-overlapping windows to prevent overfitting to a single favorable period.

Win rate, alpha-positive rate, and cumulative alpha are tracked per commit. The entire calibration history is event-sourced through Git: every commit triggers a post-commit hook that runs the backtest, records the results in calibration.json, and updates the GitHub Pages dashboard. If a code change causes the agent to regress, the dashboard shows it immediately.

Act IV

The Dashboard

The live dashboard on GitHub Pages is the agent's public ledger. It shows a performance grade (A+ through F) based on rolling alpha, alongside key metrics: latest alpha percentage, win rate, alpha-positive rate, and total calibration runs. An alpha trend chart tracks performance across commits with green and red fill areas above and below zero.

The calibration history tab is the most interesting part. It displays every backtest result chronologically, tagged with the Git commit hash that triggered it. Each entry shows the commit message, the performance metrics, and whether the change improved or regressed the agent. The dashboard is not a marketing page. It is an audit trail rendered as HTML.

The agent's track record is version-controlled.
Every improvement is a commit. Every regression is visible.

Act V

The Discipline

The project is not about whether Bitcoin will go up or down. It's about whether an AI agent can be held to the same standards as a human trader: transparent reasoning, auditable decisions, backtested performance, and honest reporting of losses. The bearish thesis is a constraint that makes the system interesting—it forces the agent to find opportunity in a market that broadly trends upward, which means it has to be right about timing and risk management.

The architecture choices reflect this discipline. SQLite stores every decision. Git hooks enforce regression testing. The dashboard publishes results automatically—good or bad. There is no cherry-picking of favorable periods. The sliding-window backtest prevents it structurally.

The most telling design choice: the agent runs on testnet by default. Going live requires explicit configuration (DERIBIT_LIVE=true) and the README carries a risk warning about unlimited loss on short positions. The system is designed to prove competence before deploying capital. That's the real thesis.
